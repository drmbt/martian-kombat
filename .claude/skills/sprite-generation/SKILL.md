# Sprite generation — prompting to minimize errors

**The prompt craft lives in ONE place: `tools/core/prompts.mjs` (FRAME_RULES,
STYLE_ART, canonical/portrait/defeat builders) + `tools/core/cells.mjs` (the
shared CELLS pose library, LOW/LYING geometric constraints, buildJobs).** Both
front doors compose from it: the CLI (`tools/gen-frames.mjs` + per-character
poses in `tools/frames-manifest.mjs`) and the Character Creator
(`src/ui/creatorModel.ts`). Improve a rule there and every future generation —
canon or creator — gets it; do NOT re-state prompt craft in per-character
poses or in this skill.

Frames are generated by `tools/gen-frames.mjs` (`gemini-3-pro-image`). The
shared `FRAME_RULES` and `STYLE_ART` are prepended automatically — a pose
prompt only needs to describe the pose + any FX. The model anchors HARD to the
canonical reference; most failures are it copying the canonical when the pose
needs to break from it. Every rule below exists because we hit the failure in
production.

**Reference-chaining policy (locked 2026-07-08, docs/CHARACTER_STUDIO.md §2.9):**
quality comes from the RIGHT reference images, not QA-reroll loops. Canonical
first (gate: arms NOT extended far forward — it kills punch variety), then
crouch/jump anchors early (they ref their whole move families), idle-b refs
idle-a (not the canonical), walk-b refs walk-a, specials chain sequentially
(startup←idle; projectile alone; active←startup+projectile+idle;
recovery←active+idle). One re-roll per asset, max.

## Universal invariants (already in FRAME_RULES — reinforce when a cell fails)

- **Exactly ONE figure.** One person, drawn once. Never a second copy, never an
  opponent, never a clone.
- **Correct anatomy:** two arms, two legs, every limb attached — no extra,
  floating, or duplicated limbs.
- **Faces RIGHT** (P1 perspective). Forward = toward the right frame edge.
- **Everything inside the frame** with a green margin on all four sides. Any
  alpha touching an edge becomes a hard keyed line — it fails QA.
- **Background is exactly flat chroma green `#00B140`**, uniform, no shadow/floor.

## Failure mode → prompt antidote

| Symptom | Antidote to add to the pose text |
|---|---|
| Crouch / lying poses drawn at standing height | This is auto-handled: `gen-frames` passes a low-pose anchor cell as a 2nd reference for crouch cells + "copy the BODY HEIGHT of the second reference." Keep the two-phase gen order (anchor first). |
| **Walk frames look like the idle stance** (no step) | `MID-STEP walking, caught in motion; UPRIGHT on a NARROW base; one knee lifted HIGH and swung forward, foot off the ground; the other leg trailing. IGNORE the wide lunging leg stance of the reference — redraw the LEGS as a walk.` The two walk frames must mirror (opposite lead leg) and differ from idle. |
| **Kick spawns a third leg** (canonical pose + extra kicking leg) | `EXACTLY ONE figure, ONE kicking leg clearly attached at the hip, no extra legs.` Name which leg: `her RIGHT leg extended, LEFT planted.` |
| Punch has no reach (barely past guard) | `the striking arm FULLY EXTENDED forward past the guard at <height>; clear reach and contrast with the idle stance.` |
| **Throw/grab draws a second person** | `she is COMPLETELY ALONE, a single figure grabbing the EMPTY GREEN AIR toward the right edge, a small impact flash at her hands — absolutely no second person, no opponent, no clone.` |
| Prop vanishes between frames (staff, spliff, cigarette) | put it in the character spec's `always:` field so it rides EVERY prompt: `She is ALWAYS holding X — it must be visible in this frame.` |
| Crouch not actually low | `squatting EXTREMELY low, hips at heel height, the entire figure in the BOTTOM HALF of the frame, empty green above.` |
| `down` (KO) wrong orientation | `lying FLAT on the back, a HORIZONTAL shape along the BOTTOM QUARTER, HEAD ON THE LEFT (fell backward while facing right), limbs sprawled.` |
| block reads like idle | `braced defensive parry — forearms crossed high, weight shifted back, clearly NOT the neutral stance.` |

## FX must be KEYABLE — never green

FX rendered in green (glow, runes, vines, flame) is unkeyable on the green
screen and vanishes when packed. Recolor to the character's kit color:
violet / silver-white / blue / gold / amber. State it explicitly and redundantly
in the pose: `moonlit-silver glow (SILVER-WHITE, never green)`. If the lore says
green (matrix runes, green portal vines), pick the nearest keyable substitute
(crimson runes, gold vines) and note WHY in a code comment.

## Specials — sequential, cross-referenced, projectile-first

`gen-frames` generates named specials SEQUENTIALLY (not pooled) so each phase
can reference the earlier ones; keep it that way. Practical consequences for
prompts:

- Write startup → active → recovery as a continuous motion; the model sees the
  prior phases, so describe the *change* ("the arm now in follow-through after
  the throw"), not a fresh pose.
- **Projectile-first:** a projectile-bearing special generates its projectile
  before the phases, so the active frame can show a matching object leaving the
  hand. Describe the released object consistently with the projectile prompt.
- **Projectile prompts reference ONLY explicit inspo images (`refPaths`), never
  the canonical** — the canonical drags the whole character into the projectile.
  Use `ONLY a <object>, a SMALL ISOLATED OBJECT floating alone, wide empty green
  margin on all sides. ABSOLUTELY NO PERSON, no body, no hands.`
- Per-move **inspo images** (a real portal, a real prop) go in
  `spec.extra.specialRefs[moveId]` and are passed to that special's phases.

## Batching hint

Idle + walk read as one loop — author them together. When re-rolling a failed
cell use `gen:frames --char <id> --cells a,b` (targeted, cheap). Always re-run
the sprite-qa validation on the raw frames before packing.

## Style / lore inputs

The locked style prompt lives in `tools/style.md` (painted cel, approved
2026-07-01) — never drift from it. Character flavor (outfit, colors, props, kit
color) comes from the **Martian Lore sheet** (see the new-character skill for the
fuzzy-search workflow) and `docs/CHARACTERS.md`.
